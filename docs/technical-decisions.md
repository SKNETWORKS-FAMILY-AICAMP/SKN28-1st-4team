# 기술 결정

## 1. 왜 `uv` workspace 인가

이 저장소는 Python 서비스가 여러 개다. 단일 패키지 구조보다 workspace가 더 적합한 이유는 다음과 같다.

- 서비스별 의존성을 분리할 수 있다.
- 루트에서 전체 동기화가 가능하다.
- 모노레포에서 공통 운영 규칙을 유지하기 쉽다.
- Dockerfile에서도 `uv sync` 흐름을 일관되게 가져갈 수 있다.

즉, `fe`, `be`, `predict_engine_host`, `predict_engine_research`, `data_collection` 을 한 저장소에서 관리하되, 각자의 런타임 성격은 분리한다.

## 2. 왜 cohort-based 모델 설계인가

동일 차량 단위 추적이 완벽하지 않은 중고차 데이터에서는 완전한 VIN 매칭만 전제로 모델을 짜면 학습 데이터가 급격히 줄어든다. 그래서 이 프로젝트는 코호트 기준선을 먼저 두고 편차를 설명하는 설계를 택한다.

이 설계의 장점은 다음과 같다.

- 데이터 결손과 중복에 더 강하다.
- 동급 대비 설명을 만들기 쉽다.
- 가격과 감가를 같은 프레임 안에서 해석하기 쉽다.
- 실제 제품 UI에 쓰기 좋은 구조다.

## 3. 왜 별도 모델 호스팅 계층을 두는가

`predict_engine_host` 를 분리한 이유는 모델 실행과 일반 API를 분리하기 위해서다.

- 모델 로딩 실패와 일반 API 실패를 다른 장애로 볼 수 있다.
- ONNX runtime 설정을 API 서비스와 독립적으로 조정할 수 있다.
- 백엔드는 예측 결과를 소비하는 계층이 되고, 모델 서비스는 예측 실행에 집중한다.

## 4. 왜 ONNX + gRPC 인가

### ONNX

- 학습 프레임워크와 추론 프레임워크를 느슨하게 분리할 수 있다.
- 배포 환경에서 PyTorch 전체 스택 의존성을 줄일 수 있다.
- 추후 다른 런타임으로 이동할 여지가 생긴다.

### gRPC

- 모델 입력/출력 계약을 `proto/` 에 명시적으로 둘 수 있다.
- 백엔드와 모델 호스팅의 책임 경계를 분명하게 나눌 수 있다.
- 추후 다중 모델 서비스나 원격 추론 환경으로 확장하기 쉽다.

## 5. 왜 `.env.sample` 을 두는가

실제 `.env` 는 저장소에 올리지 않는다. 대신 `.env.sample` 을 제공하는 이유는 다음과 같다.

- 팀원이 필요한 값의 형태를 빠르게 이해할 수 있다.
- 환경변수 누락으로 인한 초기 실행 실패를 줄일 수 있다.
- DB 접속 정보나 API 키 같은 민감 정보는 Git과 분리할 수 있다.

루트 `.gitignore` 에는 `.env`, `*/.env` 가 이미 포함되어 있다.

## 6. 왜 W&B 를 문서 설명용으로만 연결하는가

현재 요구사항은 사용자가 실험 대시보드와 상호작용하는 서비스가 아니다. 그래서 W&B는 "실험 추적 저장소" 역할로 쓰고, GitHub Pages는 "프로젝트 설명 문서" 역할로 나눈다.

이렇게 나누면 장점이 있다.

- 학습 중 지표는 W&B에 잘 기록된다.
- 공개용 문서는 GitHub Pages에서 간결하게 관리할 수 있다.
- public report만 embed 하면 Pages 쪽에 API 키가 전혀 필요 없다.
- 민감한 키를 repo에 넣지 않아도 된다.

즉, 학습 추적은 W&B가 담당하고, 설명은 Pages가 담당하는 구조다.
